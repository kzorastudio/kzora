# خطة العمل: نظام طلبيات الموظفين والطباعة الجماعية للطلبات (Staff Orders & Bulk Printing)

**تاريخ التحديث:** 2026-06-01 (مُراجَعة v2)
**المشروع:** Kzora
**الحالة:** تم التوافق عليها ومستعدة للتنفيذ

> **ملاحظات المراجعة (v2):** أُضيفت بنود أمنية حرجة لتأمين المسارات القديمة، وتوحيد منطق الإنشاء بدل تكراره، وقرارات حاسمة حول التسعير والخصومات. النقاط المُضافة موسومة بـ **[v2]**.

### 📐 قرارات حاسمة (محسومة)
- **التسعير:** سعر طلبية الموظف يُؤخذ **إجبارياً من قاعدة البيانات** (نفس سلوك متجر العملاء). لا يوجد إدخال سعر يدوي.
- **الخصومات:** طلبيات الموظفين **بدون أي خصومات إطلاقاً** — لا نقاط ولاء، ولا كوبونات، ولا خصم تعدد القطع (`multi_discount`). الإجمالي = مجموع الأسعار من قاعدة البيانات + رسوم الشحن فقط.

---

## 📌 الأهداف العامة للخطة الجديدة

1. **إتاحة إنشاء طلبيات يدوية (للموظفين والمدراء):**
   - يمكن للموظف أو الأدمن الدخول لقسم وإنشاء طلبات لعملاء كأنه عميل عادي.
   - **تخطي نظام الولاء:** هذه الطلبيات لا تُنشئ نقاط ولاء ولا تقبل خصومات الولاء التراكمية.
   - **الصلاحيات والعزل:**
     - الموظف العادي (`employee`) يرى فقط الطلبيات التي قام بإنشائها بنفسه (عزل تام).
     - المدير العام (`super_admin`) يرى جميع الطلبيات (العادية وطلبيات الموظفين) في شاشة موحدة، مع إمكانية الفلترة وتتبع أداء الموظفين.

2. **نظام الطباعة الجماعية الذكي (Bulk Printing):**
   - تحديد طلبات متعددة بالـ Checkbox وطباعتها دفعة واحدة بحجم A4.
   - تتبع حالة الطباعة عبر حقل `printed` (مطبوع / غير مطبوع).
   - **شاشة تأكيد الطباعة:** لتجنب الأخطاء البرمجية الناتجة عن تراجع الأدمن عن الطباعة في المتصفح، تفتح شاشة تجهيز مستقلة ولا يتم تحديث حالة الطباعة في قاعدة البيانات إلا بعد ضغط الأدمن على زر "تأكيد نجاح الطباعة" يدوياً في لوحة التحكم.

## 💾 1. تعديلات قاعدة البيانات (Supabase Database SQL)

سنقوم بتشغيل الاستعلامات التالية في Supabase لإضافة الحقول اللازمة لجدول `orders`:

```sql
-- 1. ربط الطلب بالموظف أو المدير الذي أنشأه
ALTER TABLE orders 
ADD COLUMN created_by_admin_id UUID REFERENCES admins(id) ON DELETE SET NULL;

-- 2. حقل لمعرفة هل تم طباعة الإيصال أم لا
ALTER TABLE orders 
ADD COLUMN printed BOOLEAN DEFAULT FALSE NOT NULL;

-- 3. تتبع وقت الطباعة
ALTER TABLE orders 
ADD COLUMN printed_at TIMESTAMPTZ;

-- 4. تتبع المستخدم الذي قام بالطباعة
ALTER TABLE orders 
ADD COLUMN printed_by_id UUID REFERENCES admins(id) ON DELETE SET NULL;

-- 5. [v2] فهرس لتسريع فلترة طلبيات الموظف والطباعة
CREATE INDEX IF NOT EXISTS idx_orders_created_by_admin ON orders(created_by_admin_id);
CREATE INDEX IF NOT EXISTS idx_orders_printed ON orders(printed);
```

> **[v2] تحديث الأنواع (TypeScript):** يجب إضافة الحقول الأربعة الجديدة إلى `interface Order` في `types/index.ts` (`created_by_admin_id: string | null`, `printed: boolean`, `printed_at: string | null`, `printed_by_id: string | null`)، وإلا سيفشل البناء.

## 🔒 0. [v2] تأمين المسارات القائمة (بند حرج — أولوية قصوى)

> العزل الحقيقي يُفرض في الـ API لا في الواجهة فقط. حالياً `getAuthSession` تتحقق من *وجود* جلسة فقط ولا تفحص الدور، فأي موظف يستطيع استدعاء `GET /api/orders` لرؤية كل الطلبات أو `DELETE /api/orders/[id]` لحذف أي طلب. يجب إصلاح هذا قبل إطلاق الميزة.

1. **`GET /api/orders` (`app/api/orders/route.ts`):**
   - إذا كان الدور `employee` → حصر النتائج بـ `created_by_admin_id = session.id` إجبارياً (تجاهل أي محاولة لتجاوز الفلتر).
   - إذا كان `super_admin` → كل الطلبات (السلوك الحالي).
2. **`PUT` و `DELETE` في `app/api/orders/[id]/route.ts`:**
   - قبل التعديل/الحذف، جلب `created_by_admin_id` للطلب.
   - إذا كان الدور `employee` ولا يملك الطلب (`created_by_admin_id !== session.id`) → إرجاع `403 Forbidden`.
   - الحذف (`DELETE`) يُفضّل قصره على `super_admin` فقط.
3. **إضافة helper مشترك** مثل `requireRole(session, 'super_admin')` لتوحيد فحص الصلاحيات عبر المسارات.

## ⚙️ 2. مسارات الـ API الجديدة والمعدلة

### أ. مسار طلبيات الموظفين (`/api/admin/staff-orders`)

> **[v2] تجنّب تكرار الكود:** منطق POST الحالي في `app/api/orders/route.ts` ضخم (~480 سطر: تحقق المخزون، الـ variants، خصم الكمية، رقم الطلب، سجل الحالة). **لا يُنسخ** هذا المنطق. بدلاً من ذلك يُستخرج المشترك في helper واحد مثل `createOrderCore(payload, options)` حيث `options = { createdByAdminId, skipLoyalty: true, skipDiscounts: true }`، يستدعيه المساران (المتجر العام ومسار الموظفين). هذا يقلل التكرار ويمنع تباعد المنطق مستقبلاً.

- **`POST`**:
  - يتطلب تسجيل دخول (Session) بدور `employee` أو `super_admin`.
  - يقوم بإنشاء الطلب وتخزينه في جدول `orders` مع تعيين حقل `created_by_admin_id` بمعرف الموظف الحالي.
  - **التسعير [v2]:** الأسعار تُؤخذ إجبارياً من قاعدة البيانات (لا سعر يدوي).
  - **الخصومات [v2]:** يتخطى **كل** الخصومات: نقاط الولاء (`loyalty_points`) + الكوبونات + خصم تعدد القطع. الإجمالي = subtotal من قاعدة البيانات + رسوم الشحن فقط. وبالتالي **لا يُدرج سطر `loyalty_points` نهائياً** لهذه الطلبية (الإنشاء العام يُدرج نقطة pending — هنا يُتخطى).
  - يقوم بالتحقق وتخفيض كميات المنتجات من المخازن (نفس منطق المتجر).
  - **[v2] رقم الطلب:** استخدام prefix مميز `KZS-` لطلبيات الموظفين (بدل `KZ-`) لتسهيل التمييز البصري والفرز وتقليل احتمال التصادم مع طلبات المتجر.
- **`GET`**:
  - إذا كان دور المستخدم `employee` -> يرجع فقط الطلبات التي تحمل `created_by_admin_id = current_user_id`.
  - إذا كان دور المستخدم `super_admin` -> يرجع كافة الطلبات مع إمكانيات الفلترة:
    - فلتر حالة الطباعة: (مطبوع / غير مطبوع / الكل).
    - فلتر مصدر الطلب: (الكل / طلبيات المتجر / طلبيات موظف معين).

### ب. تحديث حالة الطباعة جماعياً (`/api/admin/orders/bulk-print`)
- **`POST`**:
  - يستقبل قائمة من الـ IDs للطلبات ويقوم بتحديثها كـ مطبوعة (`printed = true`) وتخزين `printed_at` و `printed_by_id`.
  - **[v2]** يُنفّذ كتحديث واحد ذرّي: `update orders set ... where id in (ids)` بدل حلقة تحديثات منفصلة.
  - **[v2]** مقصور على `super_admin` (الموظف العادي لا يطبع جماعياً).
  - **[v2]** العملية idempotent: إعادة وسم طلب مطبوع مسبقاً لا تسبب ضرراً.

### ج. إحصائيات الموظفين والطلبات (`/api/admin/staff-orders/stats`)
- **`GET`**: (خاص بالمدير العام فقط)
  - يرجع قائمة بالموظفين مع ملخص أداء لكل منهم (عدد الطلبات الكلي، عدد الطلبات المسلمة والملغاة، وقيمة المبيعات الإجمالية).

## 💻 3. واجهات لوحة التحكم والصفحات الجديدة

### أ. الصفحة الرئيسية الموحدة للطلبات (`/admin/orders`)
- مخصصة للـ `super_admin` لعرض كافة الطلبات (متجر + موظفين).
- إضافة Checkboxes للتحديد المتعدد.
- فلاتر للطباعة (مطبوع / غير مطبوع) وفلاتر للموظفين.
- شريط إجراءات جماعية عند التحديد يحتوي على زر "طباعة الفواتير المحددة" وزر "تحديث الحالة جماعياً".

### ب. صفحة "طلبياتي" للموظفين (`/admin/staff-orders`)
- صفحة نظيفة للموظف العادي يرى فيها فقط طلبياته، إحصائياته الخاصة، وزر "إنشاء طلبية جديدة".

### ج. صفحة تجهيز الطباعة وتأكيدها (`/admin/orders/print`)
- تفتح في تبويب جديد بمجرد تحديد الطلبات والضغط على طباعة.
- تعرض الطلبات بشكل منسق وجاهز للطباعة (A4) وتفصل بينها فواصل صفحات الطباعة تلقائياً.
- **شريط تحكم علوي عائم (يختفي في الطباعة الورقية):**
  - زر **"🖨️ بدء الطباعة"** لتشغيل نافذة المتصفح.
  - زر **"✅ تأكيد نجاح الطباعة وحفظ الحالة"** لتحديث حالة الطلبات في قاعدة البيانات كـ مطبوعة وإعادة التوجيه.
  - زر **"❌ إلغاء وتراجع"** للعودة دون حفظ.

## 🚀 4. خطة التحقق والاختبار (Verification Plan)

1. **التحقق من العزل (واجهة + API) [v2]:** تسجيل الدخول بحساب موظف والتحقق من عدم رؤية طلبيات الآخرين في الواجهة، **و** استدعاء `GET /api/orders` مباشرة بجلسة الموظف للتأكد أنها لا تُرجع إلا طلبياته، **و** محاولة `DELETE`/`PUT` على طلب ليس له والتأكد من رفض الطلب بـ `403`.
2. **التحقق من الخصومات [v2]:** إنشاء طلب موظف لعميل يمتلك نقاط ولاء + كود كوبون فعّال + كمية تؤهّل لخصم تعدد القطع، والتأكد أن الإجمالي = subtotal + شحن فقط، وأن أي خصم لم يُطبَّق، وأن سطر `loyalty_points` لم يُنشأ لهذه الطلبية.
3. **التحقق من حالة الطباعة:**
   - الدخول إلى صفحة الطباعة لـ 3 طلبات جديدة.
   - إلغاء أمر الطباعة في المتصفح والعودة دون الضغط على "تأكيد" -> التأكد من بقائها كـ "غير مطبوعة".
   - إتمام الطباعة والضغط على "تأكيد" -> التأكد من تغير حالتها في الجدول فوراً.
